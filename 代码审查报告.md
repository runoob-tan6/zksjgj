## 三、各问题详细分析

### P0-1: `ui_basic_data.py` 第 253 行永真条件

**现状**：
```python
changed = True
if changed:
    self.layer_idx_map[entry_id] = len(self.borehole.layers)
    self.borehole.layers.append(new_layer)
    ...
```

**修复**：直接去掉 `if changed:` 判断，或改为根据实际是否新增来设置 `changed`。

---

### P0-2: `writer.py` 第 40 行备份编码丢失

**现状**：
```python
from .parser import read_existing_text
...
backup.write_text(read_existing_text(path) or "", encoding="utf-8")
```

`read_existing_text` 尝试 UTF-8 → GBK → latin-1 解码，再以 UTF-8 写入备份。原 GBK 文件备份后变成 UTF-8。

**修复**：
```python
import shutil
shutil.copy2(str(path), str(backup))
```

---

### P0-3: `parser.py` `_depth_key` 排序异常

**现状**：
```python
def _depth_key(value: str) -> float:
    try:
        return float(value)
    except ValueError:
        return 0.0
```

**修复**：
```python
def _depth_key(value: str) -> tuple:
    try:
        return (0, float(value))
    except ValueError:
        return (1, value)  # 非数字按字符串排序，排在数字之后
```

---

### P1-4: `_on_var_change` 每次按键全量重建

**现状**：`trace_add("write", self._on_var_change)` 在每个 StringVar 变化时触发，调用 `normalized_lines()`。

**建议**：在 `_on_var_change` 中只标记 dirty，延迟到 `commit_active_edit` 或切换钻孔时才同步到模型。

---

### P1-5: `deepcopy` 快照开销

**现状**：每次 `begin_borehole_change` 调用 `BoreholeSnapshot.capture`，对 layers 列表和 tests 字典做 `deepcopy`。

**建议**：
- 延迟捕获：只在用户实际修改数据时才创建快照
- 或用 JSON 序列化代替 `deepcopy`（对于 dataclass 列表通常更快）

---

### P1-6: `TestSection.load_borehole` 销毁重建

**现状**：每次加载钻孔时 `for w in self.winfo_children(): w.destroy()`，然后重新创建所有子组件。

**建议**：已有 `refresh()` 方法只更新 Treeview 内容，`load_borehole` 应复用 `refresh` 而非重建。

---

### P2-7: Treeview 编辑逻辑重复

以下方法在 `ui_basic_data.py` 和 `ui_test_data.py` 中高度相似：

| 方法 | 行数(基础) | 行数(试验) | 相似度 |
|------|-----------|-----------|--------|
| `edit_cell` | 52行 | 77行 | ~80% |
| `start_drag` | 6行 | 6行 | 100% |
| `finish_drag` | 46行 | 57行 | ~85% |
| `show_context_menu` | 30行 | 30行 | ~95% |
| `commit_active_edit` | 13行 | 15行 | ~90% |

**建议**：抽取 `EditableTreeMixin` 或 `EditableTreeSection` 基类，约可减少 ~200 行重复代码。

---

### P2-8: `normalized_lines()` 重复计算

**现状**：以下属性 getter 都会调用 `normalized_lines()`：
- `hole_id`（间接通过 `lines`）
- `normalized_lines()` 被 `__init__`、`normalized_lines()`、各种 UI 刷新调用

每次调用都 `list(self.lines[:16])` + padding + 默认值填充。

**建议**：在 `lines` setter 中缓存 `self._normalized = self._normalize()`，getter 直接返回缓存。

---

### P2-9: 排序逻辑内联

**现状**（`writer.py` `export_layer_test_summary`）：
```python
sort_key = (-1 if suffix == "m" else 1, {
    "b": 0, "c": 1, "d": 2, "g": 3, "h": 4,
    "m": 5, "n": 6, "o": 7, "q": 8, "l": 9
}.get(suffix, 99))
rows.append((sort_key, suffix, ...))
rows.sort()
```

**建议**：分离为独立的 `sort_key_func`，或先构建 rows 再排序。
